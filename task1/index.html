Intent-matched heroes for contract signing

New users land on pdf.net from different sources with different jobs. I picked signing contracts as one intent, then designed two above-the-fold heroes that speak to that job, not a generic "PDF editor".

Audience picked

I picked the blog post about signing contracts. The visitor already has a PDF, knows what they need, and is time-sensitive. They did not come to explore 20 tools. They came to sign and return the file.

Copy choices

  • Headline — "Sign PDF contracts": Names the job they came for, not "edit PDF" or "PDF tools". Matches blog-post intent and search language around signing.
  • Subtitle — upload → sign → send back in minutes: Three-step outcome in one line. "Minutes" answers the time pressure; "send it back" implies the file goes to someone else (contract context).
  • CTA — "Choose PDF to sign": Verb + object. Not "Choose file" or "Get started" — reduces ambiguity about what happens next.
  • Privacy — "Files stay private": One line, no legalese. Contracts are sensitive; this lowers upload anxiety without a trust-badge wall.
  • Variant 2 chip — "10M+ docs, 500k users/mo": Scale proof for visitors who need reassurance before uploading. No star ratings or fake reviews.

A/B hypotheses

  1. H1 — Intent-matched copy lifts first upload. Signing-intent traffic (blog, /sign-contract, "sign PDF" keywords) will upload more often when headline + CTA say "sign" vs pdf.net's generic hero. Metric: landing_view → file_uploaded
  2. H2 — Outcome visual reduces bounce before upload. Showing a signature on the document (mini editor or card stack) vs an empty dropzone will keep visitors on page long enough to upload — they see the end state before committing a file. Metric: landing_view → bounce and time to first upload

Layout (Variant 1 centered vs Variant 2 split) is a follow-up test once copy intent is validated — see measurement section below.

Proposed heroes: sign contracts

Two layout directions for the same signing intent. Shared across both: job-specific headline, outcome-focused subtitle, "Choose PDF to sign" CTA, privacy line, and an animated preview of signing, not an empty upload box.

Variant 1 keeps a centered, editorial layout: copy on top, upload below, editor animation as the hero visual.
Variant 2 uses a horizontal split: copy, usage chip, and upload on the left; a card-stack signing animation on the right.

Design principles applied

Both variants follow the same intent-matching rules. They differ in layout and how trust and outcome are shown.

  • Reduce hero noise: one job-specific action ("Choose PDF to sign") and no competing tool grid above the fold in either layout
  • Speak their words: headlines use "sign" and "contracts", matching language from the blog post and search query. Variant 2 tightens this into a two-line headline with a clearer speed promise
  • Show the outcome: Variant 1 animates a signature inside a mini editor; Variant 2 cycles signed documents in a card stack. Both show the end state before upload
  • Reduce risk: "Files stay private" on both. Variant 2 adds a usage chip ("10M+ docs, 500k users/mo") for scale proof without review widgets
  • Fit layout to scanning: Variant 1 is centered and works when the visitor reads top-down; Variant 2 puts action on the left and outcome on the right for faster left-to-right scanning

How to measure success

Primary metric: landing → file uploaded (hero CTA or dropzone). Same for both variants.

Test design: A/B Variant 1 vs Variant 2 on signing-intent traffic (blog post, /sign-contract, or paid keywords around "sign PDF"). Hold copy intent constant; isolate layout and visual treatment.

Metric What it tells us
landing_view → file_uploaded Which layout converts better: centered editor (Variant 1) vs split + card stack (Variant 2)?
landing_view → bounce Does one layout lose signing visitors faster?
hero_cta_click → editor_opened CTA clarity in both layouts: clicks that actually start the tool
uploaded → first_sign_action Intent quality: do visitors who came to sign actually sign?
Time to first upload Whether Variant 2's left-column action layout reduces hesitation vs Variant 1's stacked layout
Scroll past hero (Variant 2 only) Whether the split layout keeps upload visible without scrolling on common viewports

Segment by referrer: blog post vs organic vs paid. Min ~500 sessions per variant before reading conversion. If Variant 2 wins on upload, validate whether the usage chip or the split layout drove the lift. Optional follow-up test with chip removed.
